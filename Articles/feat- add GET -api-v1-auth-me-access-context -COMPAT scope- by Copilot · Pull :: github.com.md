The PythonAnywhere production frontend calls the Python backend directly — PR [#28](https://github.com/hn3tdevops-jpg/Workforce-backup/pull/28)'s Node.js proxy fix doesn't cover it. This adds the missing backend endpoint.

## Endpoint

`GET /api/v1/auth/me/access-context` — requires \*\*\*\*\*\* returns a COMPAT scope built entirely from existing RBAC tables (no new permissions, no employee-link tables required).

**Response shape:**

```markdown
{
  "user_id": "...",
  "has_access": true,
  "active_scope_count": 1,
  "scopes": [{
    "effective_permissions": ["hk.rooms.read", "schedule.read", ...],
    "assignments": ["Owner"],
    "link_status": "COMPAT",
    "employment_status": "ACTIVE",
    "is_super_admin": false
  }],
  "resolved_at": "2026-05-05T05:34:12.000000+00:00"
}
```

## Logic

- No active membership for token's `business_id` → `has_access: false`, `active_scope_count: 0`, `scopes: []`
- Active membership → single COMPAT scope; `effective_permissions` from `get_effective_permission_codes()`, `assignments` from `get_effective_role_names()`
- `is_super_admin: true` only when `"*"` or `"superadmin:*"` appears in resolved permissions

## Tests (`file tests/test_auth_access_context.py`)

Five focused cases: 401 unauthenticated, 200 active member, permissions present in scope, `active_scope_count == 1`, and no scope returned for a user with a valid token but no active membership.
